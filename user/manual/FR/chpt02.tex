\section{Installation et mise en marche}
Cette seconde section du manuel d'utilisation détaille toutes les étapes nécessaires pour commencer à travailler avec Giswater, depuis les étapes antérieures à son installation jusqu'à la configuration du programme, en passant par la création de nouveaux projets réels ou servant d'exemple, afin d'essayer le fonctionnement de Giswater. 

\subsection{Pré-requis avant l'installation}
En relation avec la section 1.3, où il est suggéré l'architecture idéale du système, ici sont détaillés quelques spécificités des trois serveurs qui composent le système. 

À côté du serveur de données, la performance dépend beaucoup du nombre d'utilisateurs et du volume de données. 

Comme ordre de grandeur et pour commencer avec une des personnes qui éditent les données et entre cinq et huit personnes qui les consultent, avec un volume de données relativement petit respectant le volume de personnel dédié, un appareil de quatre noyaux et un minimum de 32 GB de RAM doit être suffisant.Avec le respect du disque dur, les deux éléments à garder à l'esprit sont la capacité et la vitesse.

Dans le cas d'une organisation plus grande il serait nécessaire d'analyser la technologie des serveurs disponibles et du type de visualisation avec lesquelles ils travaillent pour ajuster aux besoins de la réalité du service et vérifier si le système est suffisant ou si il faut le re-dimentionner.

Si l'appareil est utilisé comme un serveur de PostgreSQL (et la fonction basique de S/O), l'espace du disque occupé par la base de données n'est pas trop élevé. En outre, il est possible de savoir que le disque dur s'utilise pour d'autres fonctions en plus que du stockage des données PostgreSQL. Concrètement cette usage inclus aussi: 
\begin{itemize}
\item Des fonctions basiques de suport de S/O
\item Aide via des fichiers temporels aux processus PostgreSQL
\item Stockage de fichiers specifiques de Giswater
\end{itemize}

Nous recommandons d'assigner initialement environ 100 GB de stockage. Étant donné la vitesse d'accès du disque, ce qui est un sujet particulièrement pertinent, le fait de disposer d'un disque dur solide (SSD) peut favoriser considérablement les prestations vers les utilisateurs finaux avec un contrôle qui garantit l'accès le plus rapide possible aux données. 

Du côté du serveur web, moins de prestations sont requises, et pour cela, toujours en considérant la taille de l'entreprise comme mentionné ci-dessus, avec un appareil de deux noyaux (il est recommandé d'en avoir quatre) et un minimum de 16 GB de RAM ça devrait être suffisant. Dans tous les cas, cela dépend de la concurrence et de l'utilisation faite du côté du web. 

Ce qu'il faut souligner dans ce cas est l'importance du disque pour aller stocker les photos de terrain qui seront réalisées et qui sont gérées par une base de donnée MongoDB. Pour cela, la dimension du disque, qui ne doit pas être solide, doit être ajusté à l'utilisation pour la capture journalière de photo sur le web. 

En ce qui concerne le serveurs de fichiers et en respectant le stockage des fichiers spécifiques à Giswater, il est recommandé d'utiliser le même appareil que celui de PostgreSQL pour héberger dans un fichier, seulement en mode lecture, avec un accès partagé du réseau pour les utilisateurs normaux dans lequel ils peuvent trouver un référentiel du \textit{plugin}, les projets de QGIS originaux ainsi que les originaux des instalables. 

Dans ce répertoire SIG nous stockerons les modèles des projets QGIS (sous-fichier modèle) et aussi nous configurerons pour que ce soit la route des \textit{plugins} des clients QGIS (sous-fichier plugin) que nous installerons dans tous les appareils du client. Dans le sous-fichier \textit{plugins} nous installerons tous ces compléments qui ont de l'intérêt pour l'organisation.

En respectant les autres type de données (projets QGIS, couche de base cartographique, document adjoints, ect.) nous recommandons de ne pas utiliser le même appareil. La meilleur option est d'utiliser l'unité du réseau partagé avec laquelle on travaille sur les données ( pour profiter du système de backups organisé). 
En configurant tous les utilisateurs client de ce fichier avec les \textit{plugins} avec seulement la permission de lecture, cela garantit que tous les utilisateurs de QGIS disposent des mêmes \textit{plugins} installés et c'est beaucoup plus facile des les actualiser. 

Du côté du client les appareils doivent avoir une certaine capacité de traitement (il est recommandé des processeurs i7) avec un minimum de 18 GB de mémoire. Le système opérationnel doit être obligatoirement Windows 7 ou supérieur. Si vous souhaitez effectuer une consommation intensive des ressources avec la génération de geoprocessus de grande intensité, il est recommandé d'utiliser un appareil client avec des prestations plus hautes comme pourrait l'être une multicuore de 16-32 GB de RAM et un disque dur solide. 

\subsection{Téléchargement et configuration de PostgresSQL}
PostgreSQL est une base de donnée open source avec un énorme potentiel, qui servira pour stocker toutes les données avec lesquelles travailler dans Giswater. Grâce à son extension géospatiale PostGIS, il permet une très bonne relation avec les GIS, particulièrement avec QGIS. Cette extension contient plus de 1000 fonctions geospatiales, ce qui en fait l'un des logiciels GIS les plus puissants existants, bien que PostgreSQL ne soit pas un programme spécifique de GIS. 

Il y a différentes versions qui peuvent être téléchargées. Pour travailler avec Giswater il est recommandé de télécharger toute version postérieur à la 9.5, à partir de laquelle les programmes sont 100\% compatibles. 

Le téléchargement et l'installation est très simple et peut se faire depuis https://www.postgresql.org/download. Conjointement à la base de données, il faut installer le programme d'administration de la base de données pg Admin (https://www.pgadmin.org/download/ pour le télécharger), qui servira à gérer la base de données de manière facile et visuelle pour l'utilisateur. Grâce à pgAdmin il sera possible de modifier les tables, les visites et les règles de la base de données, ainsi que consulter toutes l'information et la gérer.

Une fois les deux programmes installés, en ouvrant pgAdmin la première chose à faire est d'ajouter une connexion.

\paragraph{Configuration basique d'un projet modèle}
Il existe de nombreux types de tableaux parmis tous ceux créés dans la section précédente. Il y en a qui sont
déjà rempli par défaut, d'autres qui seront remplis lorsque des éléments du réseau seront créés et il y a
des tableaux personnalisables par les utilisateurs, en fonction de leurs besoins et de ceux de leur réseau.
Les tableaux de base qui correspondent aux principaux éléments génériques de tout projet Giswater sont d'une grande importance et doivent être pris en compte pour expliquer cette section. Ce sont les suivants:
\begin{itemize}
    \item Arc
    \item Node
    \item Connec
    \item Gully (seulement pour UD)
\end{itemize}
Ces quatre tableaux sont vides et ne seront remplis que lorsque le projet aura des éléments géospatiaux du
type correspondant. L'important est qu'ils ont de nombreuses restrictions et relations avec d'autres
tableaux qui doivent être remplis.

Voici les tableaux qui, depuis le début, doivent être remplis par l'utilisateur pour que Giswater fonctionne correctement.

\begin{enumerate}
    \item Création d'éléments de réseau nouveaux et personnalisés
    
\textbf{cat\_feature \\
node\_type \\
arc\_type \\
connec\_type \\
gully\_type (only for UD)\\}

Ces tableaux sont utilisés comme catalogue intermédiaire pour tous les éléments des différents types.
Chaque élément créé doit nécessairement faire partie d'un type d'élément (feature\_type), qui peut
être nœud, arc, connec ou gully, et en plus d'un autre type dans chacun des ci-dessus, spécifié dans
la table sys\_feature\_type (non modifiable par les utilisateurs). Il y a 24 enregistrements pour les projets d'approvisionnement en eau
(ws) et 17 pour l'assainissement (ud).

Chaque utilisateur peut personnaliser la table cat\_feature avec des éléments spécifiques nécessaires à son réseau, à condition qu'ils soient liés à l'un des champs sys\_feature\_type (par exemple, TANK ou
FONTAINE) et l'un des feature\_type existants (par exemple, NODE ou CONNEC).

Les tableaux suivants qui doivent être remplis sont ceux des types d'éléments liés à un seul feature\_type (node\_type, arc\_type, connec\_type et gully\_type). L'ID de la table cat\_feature doit correspondre à l'ID des tables * \_type, qui, bien entendu, n'auront que des éléments de leur type spécifique. Les tables * \_type
se connectent aux tables de gestion individuelles de chaque type d'élément, aussi bien que celles qui sont en action sur les modèles hydrauliques (inp).

Pour terminer la hiérarchie des tables qui déterminent les éléments existants, les tables de catalogue doivent être rempli en reliant chaque nouvel élément, avec un nom au goût de l'utilisateur, avec un ny des types spécifiés
dans les tableaux de hiérarchie supérieure (voir section 4.2.1.1
\\
    \item Domaines de valeur personnalisés
    
\textbf{man\_type\_category \\
man\_type\_fluid\\
man\_type\_function\\
man\_type\_location\\} 
Dans ces tableaux, les informations spécifiques sur le type de catégorie, de fluide, de fonction ou d'emplacement liés aux les différents types d'éléments de base (arc, connexion de nœud, gully) sont définis. Chacune des tables ont les champs suivants:

Pour donner un exemple d'utilisation de ces tableaux, nous pourrions définir un catalogue de domaine de valeurs personnalisées pour un réservoir en plus de ce qui lui correspondrait par nœud.
Ces champs sont gérés par la base de données via des clés étrangères pour garantir la cohérence et l'unicité des informations. La clé étrangère a des caractéristiques spéciales pour régir ce système de valeurs différentes selon la table source. L'une des propriétés les plus importantes est la duplicité de la clé étrangère. Cela signifie que, pour donner un exemple pour la table des nœuds, et en ce qui concerne le type de fluide, la double clé est gérée de sorte que seules les valeurs de fluid\_type dans la table man\_fluid\_type qui remplissent la condition d'avoir le champ feature\_type = 'NODE' seront disponibles dans le champ node.fluid\_type.
La double clé étrangère garantit la cohérence des informations à tout moment, évitant les insertions non conformes à ce critère et généralisant les changements en cas de renouvellement ou d'élimination des domaines de valeur.

    \item Création d'attributs personnalisés
\textbf{man\_addfields\_parameter} \\
Cette table, comme son nom l'indique, est destinée à ajouter des champs à tout élément de projet non créé
par défaut dans Giswater. Cela permet aux utilisateurs de personnaliser les champs selon ses exigences. De cette façon, à tout moment, s'il est nécessaire de lier tout type d'informations à un élément, il est possible de le faire via ce tableau.
Il existe trois autres tableaux liés à man\_addfields\_parametre.
\begin{itemize}
    \item man\_addfields\_cat\_datatype : catalogue des types de données(entier, texte, boolean)
    \item man\_addfields\_cat\_widgettype : catalogue des types de widget types dprésentés dans le formulaire (QCheckBox,QComboBox, QTextEdit ...)
    \item man\_addfields\_value : table où les valeurs sont stockées pour chaque type de paramètre ajoutés par l'utilisateur.
\end{itemize}
Les informations contenues dans les champs de ces tables ne peuvent pas être affichées dans la table attributaire de l'élément, mais il sera vu dans le fomulaire correspondant et dans les tables addfields.
\\
    \item Configuration des fonctionnalités de visite, d'inspection et de planification.\\
\textbf{om\_visit\_parameter} \\
La table om\_visit\_parameter permet d'ajouter des champs pour chaque type d'inspection qui peut être fait dans le
réseau d'eau. Tout type de visite peut être défini (par exemple, inspection ou réhabilitation) pour répondre aux
besoins de l'utilisateur. Cependant, la visite doit également être liée à un type d'élément existant.  \\
\textbf{plan\_psector} \\
Il est nécessaire d'insérer au moins un secteur de planification avant de commencer à travailler. Cette zone fonctionnera comme secteur par défaut pour les nouveaux éléments panifiés.
\end{enumerate}

\subsection{Création de projet à partir de zéro}
Si ce que nous voulons est de démarrer notre projet à partir de zéro, avec les données réelles de notre réseau, nous devons comptez les couches vitales pour qu'au moins le plugin Giswater soit activé une fois que nous sommes à l'intérieur du projet QGIS.
Les couches essentielles requises par le plugin sont celles qui appellent les vues:
\begin{itemize}
    \item \textbf{v\_edit\_arc} vue de tous les éléments de la table arc. 
    \item\textbf{v\_edit\_node} vue de tous les éléments de la table node.
    \item\textbf{v\_edit\_connec} vue de tous les éléments de la table connec.
    \item\textbf{v\_edit\_gully} (seulement UD) vue de tous les éléments de la table gully.
\end{itemize}

De plus, les tableaux listés doivent être chargés:
\begin{itemize}
\item \textbf{version} table qui stocke les informations sur les différentes versions des programmes (G iswater y
PostgreSQL), la date de création du schéma, la langue par défaut ou EPSG.
\item \textbf{exploitation} valeur d'exploitation à laquelle appartient le réseau. Il devrait être défini au moins une exploitation pour que le plugin s'active. Il est également nécessaire de filtrer les données projet utilisant l'expl\_id. 
\end{itemize}
Une fois l'information se trouve dans les tables de la base de données créée manual\_ud), vous pouvez créer un
projet QGIS et visualiser enfin l'information dans un logiciel spécifique de systèmes d'information géographique.
Dans les préférences du projet de Giswater et avec la connexion ouverte à manual\_ud, cliquez directement sur le bouton 'Créer un projet SIG'. Cela ouvrira le menu indiqué ci-dessous où les paramètres suivants sont
configuré: emplacement où le fichier QGIS sera enregistré, nom du fichier, type de projet (choisissez entre UD et WS) et le schéma de base de données avec lequel il sera lié.
Une fois tous les paramètres définis, le projet sera créé. Le projet QGIS relie toutes les tables et vues du modèle de données créé dans la section 2.3.4 avec les informations renseignées au début de cette section.
